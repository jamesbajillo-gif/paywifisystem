Personal Data Breach Notification Procedure

PAYWIFI / the Operator — RA 10173 & NPC Circular 16-03 Compliance

What is a Personal Data Breach?

Any incident that results in accidental or unlawful destruction, loss, alteration, unauthorized disclosure of, or access to personal data stored or processed by PAYWIFI. This includes: database compromise, unauthorized admin access, lost backup media, ransomware encrypting the DB, or API vulnerabilities exposing session data.

Mandatory Timelines

72 hours — Notify the National Privacy Commission (NPC) from the time of discovery, if the breach is likely to give rise to a real risk of serious harm to data subjects.

Without undue delay — Notify affected data subjects once a breach has been confirmed and the scope understood, unless notification would impede a criminal investigation.

Step-by-Step Response

Step 1 — Contain
Immediately isolate the affected system. If the API or database is compromised: systemctl stop paywifi-api paywifi-sessiond. Block external access at the firewall if needed.
Step 2 — Assess
Determine: what data was accessed/exposed, how many data subjects are affected, the likely cause, and whether harm is likely. Check logs: /var/log/paywifi/*.log and nginx access logs.
Step 3 — Preserve Evidence
Make a copy of all relevant logs before cleaning. Run sqlite3 /var/lib/paywifi/paywifi.db ".backup '/root/breach-evidence-$(date +%F).db'". Do not delete logs.
Step 4 — Notify NPC (within 72 hours)
File a breach notification at privacy.gov.ph using the NPC Breach Notification Form. Include: nature of breach, data categories affected, estimated number of data subjects, likely consequences, measures taken.
Step 5 — Notify Data Subjects
Contact affected individuals (via SMS if phone numbers are known, or via posted notice on the portal). Describe what happened, what data was involved, what steps they should take, and how to contact us.
Step 6 — Remediate
Patch the vulnerability, rotate secrets (openssl rand -hex 32 for JWT/session secrets, update API keys), restore from a clean backup, and audit access logs for unauthorized activity.
Step 7 — Document
Create an incident report: date/time discovered, nature of breach, evidence collected, notifications sent (with timestamps), and corrective actions taken. Retain for 5 years.

Severity Assessment Guide

  • High severity (NPC notification required): database dump exposed, admin credentials compromised, payment data leaked, phone numbers disclosed to unauthorized party.
  • Medium severity (document & monitor): single session record accessed without authorization, brute-force attempt with no confirmed access.
  • Low severity (document only): failed intrusion attempt, log anomaly with no data access confirmed.
